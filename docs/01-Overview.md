# PMServices — Overview & Documentation Rules (Docs-as-Code Constitution)

Этот документ задаёт правила и стандарты для документации PMServices:
- как устроена документация и как её читать;
- где находится “точка истины” (Single Source of Truth) по каждой теме;
- как добавлять/обновлять документы, диаграммы и решения (ADR);
- какие проверки должны проходить изменения (review checklist / CI quality gates).

> Документ намеренно подробный: его задача — предотвратить хаос через 6–12+ месяцев и сделать так, чтобы проект можно было сопровождать без “знаний в голове”.

---

## 0. Контекст: что такое PMServices в одном абзаце

PMServices — система управления проектами с модульной архитектурой (например: IAM, Organizations, Projects), бэкендом на ASP.NET Core WebAPI (.NET 9/10), PostgreSQL как основной БД, Redis для кэша/стримов/инвалидаций, Keycloak как сервер идентификации, Docker Compose для локального окружения (позже Aspire), Angular как SPA фронтенд, с перспективой централизованного логирования/наблюдаемости (ELK).

---

## 1. Принципы и цели документации

### 1.1 Зачем нужна документация такого уровня
Документация PMServices должна позволять:
- поднять проект локально и на препроде без устных пояснений;
- понять архитектуру и границы модулей (особенно IAM/RBAC и модель tenancy);
- безопасно внедрять изменения (контракты API, БД, инфраструктура, фронт);
- сохранять причины решений (через ADR), чтобы не “переизобретать” заново.

### 1.2 Базовые принципы
1) **Single Source of Truth**: одна тема → один главный документ-источник.  
2) **Docs follow code**: изменения в коде/инфре/контрактах обязаны отражаться в доках.  
3) **Спецификация, а не заметка**: документ — это инструкция/контракт/регламент.  
4) **Навигация — обязательна**: новые документы не живут “сами по себе”.  
5) **Нулевая толерантность к противоречиям**: лучше ссылка на источник, чем копипаст.  
6) **Коммит-история важнее “версий в названии”**: версии живут в Git, не в `*_v2_final.md`.

## 3. Single Source of Truth (точки истины)

**Single Source of Truth** (единственный источник истины) — правило:
- **что это:** у каждой темы есть один документ, где информация считается окончательной;
- **зачем:** чтобы не было расхождений между “архитектура говорит одно”, “модуль говорит другое”;
- **контекст PMServices:** много контуров и зависимостей, без этого правило проект быстро деградирует.

### 3.1 Карта точек истины
| Тема | Главный источник | Допустимы выдержки (как) | Запрещено |
|---|---|---|---|
| Навигация docs и пути чтения | `00-Index.md` | только ссылки | копировать “карту docs” в другие документы |
| Правила docs-as-code | `01-Overview.md` | краткая ссылка из индексов | “локальные правила” внутри модулей |
| Термины/аббревиатуры | `02-Glossary.md` | при первом упоминании — ссылка | вводить другие определения без согласования |
| Архитектура C4 | `03-Architecture/01..02` | модульные доки: ссылка и контекст | заново описывать контейнеры в модуле |
| Auth/IAM/RBAC | `03-Architecture/03-*` + `03-Modules/IAM/*` | FE/Infra: как потребители | дублировать RBAC-матрицы везде |
| Data Model (общая) | `03-Architecture/04-*` | модульные DB-Schema дополняют модульной частью | “вторая модель данных” в случайном месте |
| API Contracts (общие правила) | `03-Architecture/05-*` | модульные Endpoints содержат детали | два разных описания одного эндпоинта |
| Observability/Logging | `03-Architecture/06-*` | Runbook ссылается | писать политику логов в 4 местах |
| Локальный запуск | `04-Infra/01-*` | README репо: короткая ссылка | “ещё одна инструкция запуска” |
| Env/конфиги | `04-Infra/02-*` | FE Env описывает только фронтовые переменные | секреты/пароли в документации |
| Preprod deploy | `04-Infra/03-*` | Runbook ссылается | устные “шаги деплоя” без фиксации |
| Runbook | `04-Infra/04-*` | только ссылки | копировать runbook в архитектуру |
| Frontend | `05-Frontend/*` | Architecture/Modules: ссылки | дублировать auth-flow в Infra |
| Design system | `06-Design/*` | Frontend: ссылки и применимость | хранить токены в голове/чатах |
| Решения | `90-ADR/*` | документы ссылаются на ADR | “почему так” без фиксации |

### 3.2 Правило “ссылка вместо копипаста”
Если информация уже описана в источнике истины:
- в другом документе: 2–5 строк контекста + ссылка;
- таблицы/контракты/модели **не копируются целиком**.

## 4. Таксономия документов (виды документов и их роль)

### 4.1 Виды
1) **Index-доки** (`00-*-Index.md`) — навигация внутри раздела.  
2) **Спецификации архитектуры** — C4, auth, data model, contracts, observability.  
3) **Модульные спецификации** — responsibilities, endpoints, db schema, scenarios.  
4) **Infra docs** — как поднять, env, deploy, runbook.  
5) **Frontend docs** — интеграция, безопасность SPA, окружения.  
6) **Design docs** — UI Kit, tokens, rules.  
7) **ADR** — фиксация решений.

### 4.2 Инвариант для “серьёзного документа”
Любой документ, влияющий на разработку/эксплуатацию, должен содержать:
- Purpose (что/зачем/контекст)
- Scope / Out of Scope
- Definitions (термины/аббревиатуры с ссылками на Glossary)
- Artifacts (таблицы/диаграммы/примеры)
- Change Triggers (когда обновлять)
- Done Criteria (когда считаем готовым)

## 5. Стандарты оформления Markdown (структура и стиль)

### 5.1 Заголовки
- Один `#` (H1) на документ.
- Уровни не пропускать: `##` → `###` → `####`.
- Заголовки должны быть “стабильными” (чтобы якоря ссылок не ломались).

### 5.2 Таблицы: обязательные типы
Для большинства документов полезны 3 таблицы:
- “что входит / не входит” (scope),
- “триггеры обновления”,
- “критерии готово”.

### 5.3 Примеры конфигов и команд
- В код-блоках указывать язык (bash/json/yaml/csharp/sql).
- Секреты не публиковать (см. раздел Security).

## 6. Правила именования (файлы, секции, артефакты)

### 6.1 Имена файлов
- Индекс: `00-<Area>-Index.md`
- Основные документы: `01-..`, `02-..` (по порядку чтения)
- Стиль: **kebab-case** и без “версий”.

Примеры:
- ✅ `03-Architecture/03-Auth-IAM-RBAC.md`
- ✅ `04-Infra/02-Env-Config.md`
- ❌ `архитектура-финал.md`, `auth_v2.md`, `new.md`

### 6.2 Имена диаграмм
Единый стандарт хранения (соответствует структуре `docs/_assets/*`):
- исходник: `docs/_assets/diagrams/<area>/<name>.puml`
- результат (для просмотра в GitHub): `docs/_assets/diagrams/<area>/<name>.svg`

Имя диаграммы должно отражать смысл:
- ✅ `_assets/diagrams/architecture/c4-context.puml`
- ✅ `_assets/diagrams/auth/login-sequence.puml`
- ✅ `_assets/diagrams/infra/preprod-deploy.puml`

### 6.3 Термины и аббревиатуры
При первом упоминании:
- расшифровать;
- коротко: что это → зачем → контекст;
- ссылка на `02-Glossary`.

## 8. Диаграммы (PlantUML): правила хранения, генерации, использования

**PlantUML** — текстовый язык диаграмм:
- **что это:** `.puml` файл с описанием → генерация `.svg/.png`;
- **зачем:** диаграммы можно review’ить как код, легко поддерживать;
- **контекст PMServices:** C4, последовательности авторизации, деплой.

### 8.1 Где хранить
- `docs/_assets/diagrams/` — диаграммы PlantUML:
  - `.puml` — **точка истины** (SSOT: Single Source of Truth — единая точка истины)
  - `.svg` — **обязательный артефакт для просмотра** (чтобы GitHub показывал диаграмму без генерации)
- `docs/_assets/images/` — остальные изображения (скриншоты, мокапы и т.п.)

Правило:
- если диаграмма упоминается в документе, то в `_assets/diagrams/...` должны быть **оба файла**: `.puml` и `.svg`.

### 8.2 Как встраивать в Markdown
Рекомендуемый формат:
- Вставляем картинку (`.svg`) и рядом даём ссылку на исходник (`.puml`) — чтобы можно было быстро править и review’ить изменения.

Пример:
- `![Auth Login Sequence](./_assets/diagrams/auth/login-sequence.svg)`
- Исходник: [login-sequence.puml](./_assets/diagrams/auth/login-sequence.puml)

### 8.3 Генерация диаграмм (рекомендуемая практика)
**CI (Continuous Integration — автоматические проверки в GitHub Actions)**:
- **что это:** пайплайн, который запускается на PR;
- **зачем:** чтобы диаграммы/ссылки/формат не ломались;
- **контекст PMServices:** особенно важно при росте команды.

Минимально:
- либо коммитим уже сгенерированные `.svg`,
- либо CI проверяет, что `.svg` соответствует `.puml` (на будущее).

> В рамках этого документа — только правила. Конкретная реализация CI описывается в `04-Infra` (или отдельном `docs/04-Infra/...` документе, если появится).

### 8.4 Типы диаграмм и где они применяются
| Тип | Когда использовать | Где обычно |
|---|---|---|
| C4 Context | граница системы и внешние акторы | `03-Architecture/01-*` |
| C4 Containers | контейнеры (API, DB, Redis, Keycloak, FE) | `03-Architecture/02-*` |
| Sequence | логин/refresh/logout/ошибки | `03-Architecture/03-*`, `05-Frontend/02..03` |
| Deployment | окружения, preprod, зависимости | `04-Infra/03-*` |
| Data/ER | таблицы/связи/ключи | `03-Architecture/04-*`, `03-Modules/*/03-*` |

## 9. Как избегать противоречий (механика)

### 9.1 “Два уровня” описания
- Архитектура отвечает на “что” и “почему” (C4, решения, правила).
- Модуль отвечает на “как реализуем в рамках модуля” (эндпоинты, таблицы, сценарии).

### 9.2 Контракты — только в одном месте
- Правила контрактов API: `03-Architecture/05-API-Contracts.md`
- Детали эндпоинтов: `03-Modules/*/02-Endpoints.md`

Если модуль пишет про эндпоинт:
- он не придумывает формат ошибок заново,
- он следует общим правилам и ссылается на них.

### 9.3 Политика “обновление в одном PR”
Если PR меняет:
- контракт API,
- схему БД,
- auth flow,
- окружение/compose/env,
то PR должен включать обновление соответствующих документов или ссылку на связанный PR.

## 10. ADR (Architectural Decision Record): фиксация решений

**ADR** — запись архитектурного решения:
- **что это:** документ “контекст → решение → последствия”;
- **зачем:** чтобы спустя время было ясно, почему сделано так;
- **контекст PMServices:** tenancy, RBAC, хранение токенов SPA, стратегия кэша, логирование.

### 10.1 Когда заводить ADR (правило)
ADR обязателен, если решение:
- влияет на безопасность (токены, RBAC, роли, хранение секретов);
- меняет модель данных/tenancy;
- меняет границы модулей и контракты между ними;
- влияет на эксплуатацию (логирование, трассировка, деплой);
- сложно/дорого откатывается.

### 10.2 Структура ADR (минимум)
- Title
- Status: Proposed / Accepted / Deprecated
- Context
- Decision
- Consequences (positive/negative)
- Alternatives considered
- Links (PR, issues, документы)

### 10.3 Связь ADR с PR/кодом
- В описании PR: ссылка на ADR.
- После merge: ADR переводится в `Accepted`.
- В changelog архитектурных документов: ссылка на ADR (если решение влияет на них).

### 10.4 Жизненный цикл ADR (PlantUML)
```plantuml
@startuml
left to right direction
skinparam linetype ortho

rectangle "Proposed" as P
rectangle "Accepted" as A
rectangle "Deprecated" as D

P --> A : принято в PR\n(ссылка на merge)
A --> D : заменено новым решением\n(ссылка на новый ADR)
@enduml
