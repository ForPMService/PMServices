## 1) Как пополнять словарь

Добавляй термин, если он:
- встречается в API/коде/документации;
- влияет на безопасность, данные или архитектуру;
- может трактоваться по-разному.

Шаблон записи:
- **Термин (RU / EN / Аббревиатура)** → *что это* → *зачем нужно* → *контекст в PMServices (где используется/на какой документ смотреть)*

## 2) Аббревиатуры и термины (A–Z)

### API — Application Programming Interface (интерфейс программирования приложений)
Что это: контракт взаимодействия между системами (эндпоинты, форматы данных, коды ошибок).  
Зачем: чтобы фронт/интеграции работали предсказуемо.  
Контекст: `docs/03-Architecture/05-API-Contracts.md`, `docs/03-Modules/*/02-Endpoints.md`.

### ADR — Architectural Decision Record (запись архитектурного решения)
Что это: короткий документ “контекст → решение → последствия”.  
Зачем: фиксирует причины выбора, облегчает изменения в будущем.  
Контекст: `docs/90-ADR/*`, правила — `docs/01-Overview.md`.

### AuthN / Authentication (аутентификация)
Что это: “кто ты?” — проверка личности (логин/пароль, SSO и т.п.).  
Зачем: чтобы выдавать токены только реальному пользователю.  
Контекст: Keycloak + сценарии `docs/03-Architecture/03-Auth-IAM-RBAC.md`, `docs/05-Frontend/02-Auth-Keycloak.md`.

### AuthZ / Authorization (авторизация)
Что это: “что тебе можно?” — проверка прав на действие/ресурс.  
Зачем: чтобы один пользователь не делал операции от имени другого.  
Контекст: IAM/RBAC, см. `docs/03-Architecture/03-Auth-IAM-RBAC.md`, модуль IAM.

### BFF — Backend For Frontend (бекенд для фронта)
Что это: отдельный серверный слой специально под нужды SPA/мобилки.  
Зачем: уменьшает риски хранения токенов в браузере, упрощает агрегацию API.  
Контекст: пока опционально (будущее масштабирование/безопасность).

### C4 — C4 Model (модель архитектурных диаграмм)
Что это: набор уровней диаграмм (Context/Container/Component/Code).  
Зачем: быстро объяснять систему на разных масштабах.  
Контекст: `docs/03-Architecture/01-Context-C4.md`, `02-Containers-C4.md`.

### CORS — Cross-Origin Resource Sharing (политика доступа между доменами)
Что это: правила браузера, разрешающие/запрещающие запросы к API с другого origin.  
Зачем: безопасность + корректная работа SPA.  
Контекст: фронт ↔ API ↔ Keycloak; настройки в API и Keycloak.

### CSRF — Cross-Site Request Forgery (межсайтовая подделка запроса)
Что это: атака, когда браузер “по привычке” отправляет куки на чужой запрос.  
Зачем: важно при cookie-based auth (особенно если появится BFF).  
Контекст: текущая модель со SPA и токенами — оценивать риски по факту выбранного хранения токенов (см. `docs/90-ADR/*` про токены).

### DDD — Domain-Driven Design (предметно-ориентированное проектирование)
Что это: подход, где доменная модель и границы модулей важнее структуры папок.  
Зачем: держать сложность под контролем по мере роста PMServices.  
Контекст: модули `docs/03-Modules/*`.

### ELK — Elasticsearch + Logstash + Kibana (стек логирования)
Что это: система хранения/поиска/визуализации логов.  
Зачем: расследование инцидентов, аудит, наблюдаемость.  
Контекст: `docs/03-Architecture/06-Observability-Logging.md`, `docs/04-Infra/*`.

### JWT — JSON Web Token (токен в формате JSON + подпись)
Что это: компактный “паспорт” с claims (полями) + криптографическая подпись.  
Зачем: API может проверять права без обращения к БД на каждый запрос (в разумных пределах).  
Контекст: Keycloak выдаёт JWT; фронт отправляет в API.

### IAM — Identity and Access Management (управление идентификацией и доступом)
Что это: подсистема пользователей, ролей, прав, политик и связей.  
Зачем: единый контроль доступа во всех модулях.  
Контекст: `docs/03-Modules/IAM/*`, `docs/03-Architecture/03-Auth-IAM-RBAC.md`.

### IdP — Identity Provider (провайдер идентификации)
Что это: система, которая аутентифицирует пользователя и выдаёт токены.  
Зачем: централизованный вход/SSO.  
Контекст: Keycloak — наш IdP.

### Keycloak (Кейклок)
Что это: open-source IdP/SSO сервер, реализующий OIDC/OAuth2.  
Зачем: логин, токены, realm/clients/roles, интеграции.  
Контекст: `docs/05-Frontend/02-Auth-Keycloak.md`, `docs/04-Infra/*`.

### OIDC — OpenID Connect (надстройка над OAuth2 для логина)
Что это: стандарт идентификации поверх OAuth2: ID Token, UserInfo и т.д.  
Зачем: “вход” (AuthN) + базовая информация о пользователе.  
Контекст: фронт ↔ Keycloak.

### OAuth2 — Open Authorization 2.0 (протокол авторизации)
Что это: стандарт выдачи токенов для доступа к ресурсам.  
Зачем: безопасная авторизация клиентов и пользователей.  
Контекст: Keycloak, SPA (обычно Authorization Code + PKCE).

### PKCE — Proof Key for Code Exchange (защита кода авторизации)
Что это: механизм, который защищает OAuth2 flow в публичных клиентах (SPA).  
Зачем: снижает риск перехвата authorization code.  
Контекст: SPA ↔ Keycloak, обычно обязателен для SPA.

### RBAC — Role-Based Access Control (контроль доступа на основе ролей)
Что это: права выдаются через роли (Role → Permissions).  
Зачем: управляемая модель доступа без “если/то” в коде повсюду.  
Контекст: `docs/03-Architecture/03-Auth-IAM-RBAC.md`, модуль IAM.

### Permission (право)
Что это: атомарное разрешение на действие (например: `projects.read`, `projects.create`).  
Зачем: гранулярность доступа и удобная проверка в API.  
Контекст: IAM + guards/policies в API.

### Scope (область доступа)
Что это: именованный набор прав или ограничений, часто используемый в токенах/политиках.  
Зачем: упрощает “пакеты” доступа.  
Контекст: зависит от выбранной модели маппинга в Keycloak/IAM.

### SPA — Single Page Application (одностраничное приложение)
Что это: фронтенд (Angular), который живёт в браузере и общается с API.  
Зачем: быстрый UI и богатое взаимодействие.  
Контекст: `docs/05-Frontend/*`.

### Tenant (тенант, арендатор)
Что это: логическая “среда/контур” внутри SaaS, где данные и доступ изолированы.  
Зачем: мультиарендность и безопасность (не смешивать данные разных компаний).  
Контекст: `docs/03-Architecture/04-Data-Model.md`, ADR про модель мультиарендности.

### Organization (организация)
Что это: бизнес-сущность “компания/команда”, обычно внутри tenant (или = tenant, зависит от модели).  
Зачем: группировка пользователей, проектов, прав.  
Контекст: `docs/03-Modules/Organizations/*`.

### Workspace (рабочее пространство)
Что это: контейнер проектов/артефактов (если введём).  
Зачем: удобная структура для больших организаций.  
Контекст: будущая эволюция модели (если понадобится).

### Redis (Редис)
Что это: in-memory хранилище ключ→значение + дополнительные структуры данных.  
Зачем: кэш, блокировки, rate limiting, очереди (в зависимости от выбранной схемы).  
Контекст: `docs/04-Infra/*`, решения по кэшу — ADR.

### Redis Streams (потоки Redis)
Что это: структура данных Redis для потоковых сообщений (append-only log внутри Redis).  
Зачем: события/очереди без отдельного брокера на ранних этапах.  
Контекст: возможная шина событий для инвалидации кэша/аудита (смотри ADR при выборе).

### Kafka (Апач Кафка)
Что это: распределённый брокер событий/лог (streaming platform).  
Зачем: высокий throughput, масштабируемость, долговременные топики и потребители.  
Контекст: “будущий уровень”, если перерастём Redis Streams по требованиям.

### PostgreSQL (Постгрес)
Что это: реляционная база данных.  
Зачем: транзакции, целостность данных, сложные запросы.  
Контекст: основная БД приложения + отдельная БД Keycloak (по инфраструктуре).

### Observability (наблюдаемость)
Что это: способность понимать “что происходит” по логам/метрикам/трейсам.  
Зачем: быстрее находить причины проблем и измерять качество.  
Контекст: `docs/03-Architecture/06-Observability-Logging.md`.

### OTEL / OpenTelemetry (ОпенТелеметри)
Что это: стандарт/набор инструментов для метрик/трейсов/логов.  
Зачем: единый формат телеметрии без привязки к вендору.  
Контекст: будущий/планируемый слой наблюдаемости.

### Runbook (ранбук)
Что это: “инструкция дежурному” — что делать при инцидентах/типовых проблемах.  
Зачем: меньше паники, быстрее восстановление.  
Контекст: `docs/04-Infra/04-Runbook.md`.

## 3) Мини-словарь Keycloak (часто путают)

### Realm (реалм)
Что это: “контур” настроек и пользователей в Keycloak (почти как отдельная система внутри).  
Зачем: изоляция окружений/клиентов/ролей.  
Контекст: инфраструктура и интеграции с фронтом/API.

### Client (клиент)
Что это: приложение, которое просит токены (SPA, API, сервис).  
Зачем: разделение настроек и flow’ов.  
Контекст: Angular — public client; API — resource server.

### Role (роль)
Что это: набор прав/политик в Keycloak (realm/client roles).  
Зачем: выдача доступа пользователям и сервисам.  
Контекст: маппинг ролей ↔ permissions определяется архитектурным решением.

## 4) Термины домена PMServices (бизнес-слой)

### Project (проект)
Что это: контейнер задач/артефактов/участников.  
Зачем: основная единица работы.  
Контекст: `docs/03-Modules/Projects/*`.

### Artifact (артефакт)
Что это: любой объект управления проектом (задача, риск, документ, релиз и т.п.).  
Зачем: унификация сущностей под будущий рост “как Jira”.  
Контекст: будет развиваться по мере добавления модулей.

---

## 5) Definition of Done (DoD) для этого словаря

Словарь “готов”, если:
- новые термины из архитектурных и модульных доков добавлены сюда;
- нет дублей с разными определениями (одна трактовка — одна запись);
- у каждого ключевого термина есть “что это → зачем → контекст”.
